# 08_review_checklists — 论文严谨性自查清单

> **跨仓库精进**：本文件覆盖逐项检查。投稿前的系统性审计方法论（五层审计优先级、声明-证据映射、LaTeX 格式审计、GB/T 7714 参考文献核验、Results 段落架构速查）见 `<LOCAL_PATH>`。

## 总体检查

- [ ] 每条核心结论都有证据（数据/引用/实验）支撑？
- [ ] 没有证据支持的推断是否标注了"可能"/"推测"？
- [ ] 方法和实验部分是否足够详细到别人能复现？
- [ ] 数据来源和预处理步骤是否清楚？
- [ ] 引用是否真实读过（不是从别人参考文献顺手抄的）？
- [ ] 图表是否每个都有解释性的 caption？
- [ ] 有没有夸大创新？（"首次"、"突破性"是否站得住？）

## 方法检查

- [ ] 核心方法是否有足够的实现细节？
- [ ] 超参数是如何选择的？
- [ ] 是否有对照组（baseline）？
- [ ] 评价指标是否恰当？（空间任务用空间交叉验证了没？）
- [ ] 随机种子是否固定？
- [ ] 是否存在数据泄露？（如用未来数据预测过去）

## GIS/遥感专项

- [ ] 坐标系/投影是否说明？
- [ ] 数据空间分辨率是否清楚？
- [ ] 时间范围是否说明？
- [ ] 训练/验证/测试区域是否空间独立？（防止空间自相关导致的过拟合）
- [ ] MAUP（可变空间单元问题）是否考虑？
- [ ] 尺度效应是否讨论？
- [ ] 地图可视化是否规范（比例尺、指北针、图例）？

## 实验检查

- [ ] 是否做了消融实验（验证每个模块的贡献）？
- [ ] 是否做了参数敏感性实验？
- [ ] 是否对比了至少 2-3 个同类方法？
- [ ] 结果是否可复现的（不是挑最好的那一次）？
- [ ] 统计显著性是否报告了（p 值/置信区间）？

## 写作检查

- [ ] 标题是否准确反映了核心贡献？
- [ ] 摘要是否独立完整（不引用正文）？
- [ ] 引言是否讲清了：背景→问题→现有不足→本文方案→贡献？
- [ ] 相关工作是否覆盖了关键文献，而非随意堆叠？
- [ ] 讨论部分是否对结果做出了解释，而非只是复述结果？
- [ ] 局限是否诚实说明了？

## 审稿人可能攻击点

1. **数据和方法的可复现性不足** — 缺数据来源/代码/参数
2. **创新不足** — 只是换了个数据集跑已有方法
3. **对比不公平** — baseline 没调好参数
4. **评价指标有偏差** — 选了对自己有利的指标
5. **空间自相关导致过拟合** — 训练/测试区域不独立
6. **结论超过证据范围** — 夸大结果
7. **与已有工作区分不清** — related work 没写清楚差异
8. **缺少局限性讨论**

## 修改优先级规则

| 优先级 | 问题类型 | 行动 |
|--------|---------|------|
| P0 | 数据泄露、错误结论、虚假数据 | 必须修改 |
| P1 | 缺少关键对比、评价偏差 | 应该修改 |
| P2 | 表达不清晰、格式不规范 | 建议修改 |
| P3 | 可加但非必须的补充实验 | 可以不改，留到 rebuttal |

## AI 失败模式检查（7 项）

来源：academic-research-skills（源自 2026 Nature 论文），每次实验/写作/审稿前必查。

| # | 失败模式 | 说明 | 判定 |
|---|---------|------|------|
| 1 | 实现 Bug 通过 AI 自审 | 代码运行无报错但结果可能错误。AI 看到输出数值"差不多"就接受了 | CLEAR / SUSPECTED / EVIDENCE |
| 2 | 虚构引用 | 引用不存在/年份错/期刊错/结论被张冠李戴 | CLEAR / SUSPECTED / EVIDENCE |
| 3 | 虚构实验结果 | AI 写了"提升了 12%"但没有实验产出这个数字 | CLEAR / SUSPECTED / EVIDENCE |
| 4 | 捷径依赖 | 模型利用了虚假特征（如背景颜色）而非真实规律 | CLEAR / SUSPECTED / EVIDENCE |
| 5 | Bug 被重构为"新发现" | 意外结果实际是 Bug，但被叙述为"有趣的发现" | CLEAR / SUSPECTED / EVIDENCE |
| 6 | 方法论编造 | Methods 写的过程与实际运行的过程不一致 | CLEAR / SUSPECTED / EVIDENCE |
| 7 | 早期 Frame-Lock | 早期错误承诺（选题方向/方法选择/超参数方向）被后续环节无条件继承 | CLEAR / SUSPECTED / EVIDENCE |

**三态判定**：
- CLEAR：已核实，无问题
- SUSPECTED：有可疑迹象，需人工复查
- INSUFFICIENT EVIDENCE：证据不足，无法判断

**阻断规则**：任一模式为 SUSPECTED 或 INSUFFICIENT EVIDENCE → 阻断推进，必须修正后重查。

## 反谄媚协议

来源：academic-research-skills — 防止 Agent 为"讨好"用户而放弃批判性。

### 反驳评分（1-5）

| 分数 | 含义 | 行动 |
|------|------|------|
| 5 | 新证据直接命中核心攻击 | 让步 |
| 4 | 显著削弱攻击，有小缺口 | 让步+标注缺口 |
| 3 | 部分相关但回避核心 | **坚持**。复述原始攻击 |
| 2 | 跑题/换话题 | **反击**。指出偏离 |
| 1 | 无证据断言/诉诸权威 | **升级**。加强原始攻击 |

### 禁止规则

1. 永不只因为用户反驳就让步。反驳不等于证据。
2. 不能连续让步。上一次让步后，下一次让步门槛升至 5/5。
3. 让步率 > 50% 时暂停：「我已经让了几次——是我太宽松了，还是反驳确实有道理？」
4. 每 3 轮反问：「这整个讨论有没有一个我没质疑过的前提？」（Frame-Lock 检测）

### 生效时机

- Agent 在审稿/检查/评估时自动应用
- 用户说「你觉得有没有问题」时激活
- 论文自检时必过反谄媚
